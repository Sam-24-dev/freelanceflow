# Account and Profile — Final implementation handoff

## Decision first

Implement **Cuenta / Perfil** as a new operational module without replacing or weakening the existing role gateway.

- The current `acceso.html` flow remains the source of truth for selecting the **operational** or **administrative** profile.
- The operational profile can open **Cuenta** and maintain three local fields: full name, email and country.
- The administrative profile continues to see **Bitácora only**.
- The registration and login compositions produced by Stitch are retained as visual references for a future authentication phase. They are **not implemented in this slice**, because FreelanceFlow currently has no backend authentication and must not store or pretend to validate passwords.
- No password, token, credential, OAuth flow, account verification or password recovery is introduced.

This boundary is intentional: it adds a useful product module now and avoids presenting a false security boundary.

## Quick implementation path

1. Create an isolated worktree from the latest `origin/main` using branch `feat/account-profile-module`.
2. Preserve the current access, role and Bitácora behavior.
3. Add `pages/cuenta.html`, a small profile model/store and its page controller.
4. Integrate Cuenta into operational navigation only.
5. Register one privacy-safe Bitácora event after a successful profile save.
6. Validate the module through real browser flows at desktop, tablet and mobile widths.
7. Run the repository checks and Gentle-AI lifecycle before requesting authorization to publish.

## Product contract

### User and role

- **Operational**: can navigate to and edit Cuenta.
- **Administrative**: cannot see Cuenta and any direct request redirects to `bitacora.html`.
- **No valid profile**: redirects to `acceso.html`.
- **Unknown or corrupt profile value**: fails closed to `acceso.html`.

### Page

- Route: `pages/cuenta.html`.
- Navigation label: `Cuenta`.
- Page title: `Mi cuenta`.
- Introductory copy: `Administrá la información básica que identifica tu perfil de trabajo.`
- Primary action: `Guardar perfil`.
- Secondary dirty-state action: `Descartar cambios`.
- Success feedback: `Perfil actualizado. Tus cambios se guardaron correctamente.`

Do not display `demo`, `simulación`, `prototipo`, `Fase 1` or implementation disclaimers in the UI.

### Fields

| Field | Type | Rules |
| --- | --- | --- |
| `fullName` | text | Required; trim outer whitespace; 2–80 characters |
| `email` | email | Required; trim; valid browser email shape; maximum 160 characters |
| `country` | text | Required; trim outer whitespace; 2–80 characters |

If the latest main already provides a shared country selector, reuse it. Otherwise use a native text input with `autocomplete="country-name"` rather than introducing an incomplete country catalog. Do not infer tax rules, locale or legal residence from this value.

### Avatar

- Generate initials from `fullName`.
- Provide a stable decorative/avatar presentation with accessible text.
- Do not upload, resize, encode or persist image files.
- Do not add camera permissions or external image services.

### Profile-first state

When no saved profile exists:

- Render empty fields without invented personal data.
- Show an initials placeholder.
- Explain what completing the profile changes: only the local Cuenta presentation.
- Keep the save action available.

Do not prefill a fictional person merely to make the screen look populated.

## Data and persistence contract

Use one versioned localStorage key:

```text
freelanceflow.profile.v1
```

Stored shape:

```json
{
  "version": 1,
  "fullName": "",
  "email": "",
  "country": ""
}
```

Rules:

- Wrap every localStorage read and write in `try/catch`.
- Parse and validate the complete object before using it.
- Ignore unknown properties.
- Reject wrong versions and wrong field types.
- Recover corrupt JSON to the profile-first state without crashing.
- Save only after all fields pass validation.
- A storage failure must preserve the edited form and show a useful error.
- Never write passwords, tokens, role values or Bitácora events into this key.

Keep the model small and functional. Reuse existing project storage and validation conventions when they already solve the same problem.

## Form behavior

1. Load and validate the saved profile.
2. Render the stored values or profile-first state.
3. Track whether the form differs from the last persisted value.
4. Validate inline on submit and focus the first invalid field.
5. Disable repeated submission only while saving.
6. Save atomically to localStorage.
7. Re-render the summary and initials from the saved value.
8. Announce success through an `aria-live` region.
9. Enable `Descartar cambios` only when the form is dirty.
10. Discard restores the last persisted value after confirmation when data would be lost.

Use native HTML validation as the first layer. Add only the JavaScript needed for product-specific messages and persistence.

## Bitácora contract

Register only one new business event:

| Trigger | Module | Action | Detail |
| --- | --- | --- | --- |
| Successful save | Cuenta | Perfil actualizado | Información básica del perfil actualizada. |

Privacy rules:

- Do not log name, email, country, initials, storage values or IDs.
- Failed validation, field typing, discard and page refresh do not create events.
- Page-entry logging remains the responsibility of the shared shell and must not duplicate.
- A failure in optional activity logging must not undo or falsely report a successful profile save.

## Accessibility contract

- One `h1`.
- Existing skip link targets the main content.
- Headings follow a logical hierarchy.
- Every control has a persistent `<label>`.
- Use `autocomplete="name"`, `autocomplete="email"` and `autocomplete="country-name"` where appropriate.
- Use `type="email"`, meaningful `name` attributes and browser-friendly input modes.
- Avatar decoration is hidden from assistive technology; the profile name remains real text.
- Validation messages use `role="alert"` or an equivalent announced association.
- Success feedback uses `aria-live="polite"`.
- The first invalid field receives focus after submit.
- Focus indicators remain clearly visible.
- Text contrast meets WCAG 2.2 AA.
- State is never communicated by color alone.
- Touch targets are at least 44 × 44 px.
- Respect `prefers-reduced-motion`.
- Browser zoom at 200% retains all essential content and actions.

## Security and privacy contract

- Render dynamic values with `textContent`, form `.value` or safe DOM APIs.
- Do not interpolate profile data into `innerHTML`.
- Validate stored JSON before rendering it.
- Do not persist or log credentials, passwords, tokens or session secrets.
- Do not create fake authentication, authorization or account verification.
- Do not make external requests for profile data.
- Do not expose personal data in Bitácora, console messages or query parameters.
- Do not accept avatar files in this slice.
- Keep role guards fail-closed.

The absence of backend authentication is an explicit scope boundary, not a defect to conceal with frontend-only credential handling.

## Performance contract

- Cuenta should require no mock-data fetch.
- No polling, timers, workers or background requests.
- Register event listeners once.
- No repeated full-shell render after each keystroke.
- Update only the local preview values needed by the form.
- Add no images, fonts or dependencies.
- Avoid layout measurement in JavaScript.

## Real-user validation matrix

### Operational happy path

1. Landing → Acceso.
2. Select operational profile.
3. Navigate to Cuenta through the real shell.
4. Confirm the profile-first state has no invented person.
5. Enter valid full name, email and country.
6. Save.
7. Confirm visible success feedback and updated initials/summary.
8. Reload.
9. Confirm persistence.
10. Open Bitácora as administrator and confirm one privacy-safe save event.

### Validation and recovery

1. Submit every required field empty.
2. Confirm field-specific messages and focus on the first error.
3. Enter malformed email.
4. Enter boundary-length names and email.
5. Create unsaved changes and discard them.
6. Create unsaved changes and attempt navigation; confirm data-loss handling.
7. Inject corrupt JSON into the profile key and reload.
8. Inject a wrong-version object and reload.
9. Simulate unavailable localStorage when the browser tooling permits it.
10. Confirm no crash, no leaked data and a recoverable state.

### Roles

1. Administrative profile requests `cuenta.html` directly → Bitácora.
2. No profile requests `cuenta.html` → Acceso.
3. Invalid/corrupt session profile requests `cuenta.html` → Acceso.
4. Operational navigation contains Cuenta once.
5. Administrative navigation does not contain Cuenta.

### Bitácora

1. Successful profile save creates exactly one event.
2. Repeated clicks during one save do not duplicate the event.
3. Validation failure creates no event.
4. Discard creates no event.
5. Event contains no name, email, country, IDs or storage values.
6. Logger failure does not invalidate an already successful save.
